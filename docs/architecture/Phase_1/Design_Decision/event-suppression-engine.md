Event Suppression Engine
Status

Accepted

Context

Security Operations Centers (SOCs) routinely process large volumes of events generated by security tools, operating systems, applications, network devices, and cloud services. A significant portion of these events represent known benign activity and do not require analyst action.

Generating alerts for every event can lead to alert fatigue, reduced analyst effectiveness, and increased risk of overlooking legitimate threats. To improve signal-to-noise ratio and support efficient security operations, SentinelOps requires a mechanism to identify and suppress known benign events before alert creation.

Decision

SentinelOps will implement an Event Suppression Engine within the Alert Service processing workflow.

The Event Suppression Engine will evaluate events before alert generation and determine whether the event should be suppressed based on predefined suppression rules.

Events that match suppression criteria will be marked as processed and retained for auditing purposes but will not generate analyst-facing alerts.

Events that do not match suppression criteria will continue through the alert evaluation process.

Initial Phase 1 Implementation

During Phase 1, suppression rules will be implemented directly within the Alert Service codebase.

This approach was selected because it:

Reduces implementation complexity
Accelerates development of the initial platform
Simplifies testing and validation
Avoids introducing additional services or database dependencies during the foundation phase

Examples of suppression conditions may include:

Approved vulnerability scanner IP addresses
Internal laboratory systems
Authorized service accounts
Scheduled maintenance activities
Known testing environments
Consequences
Positive Consequences
Reduced alert fatigue
Improved analyst efficiency
Cleaner alert dataset
Easier future migration to advanced suppression mechanisms
Negative Consequences
Rule updates require code changes
Rule updates require application redeployment
Limited flexibility for non-technical analysts
Future Considerations

Future phases may migrate suppression rules from application code to a dedicated database table or management service.

Potential future enhancements include:

Analyst-managed suppression rules
Time-based suppression windows
Asset-specific suppression rules
Threat-intelligence-aware suppression
Alert correlation
Risk-based prioritization
Machine learning assisted event classification
Rationale

The Event Suppression Engine establishes an early foundation for detection engineering practices within SentinelOps. Implementing suppression during Phase 1 creates a realistic security operations workflow while maintaining a manageable level of technical complexity.

The selected approach balances operational effectiveness with implementation simplicity and provides a clear migration path toward more advanced event processing capabilities in future phases.
